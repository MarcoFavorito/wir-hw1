Progetto numero 1.  Implementazione di un server  FTP.

L'obiettivo di tale progetto e' l'implementazione di un  client FTP e di un server  FTP multithread in grado di implementare le seguenti funzionalità:

  • get: comando che permette di trasferire un file remoto nella directory locale al Client (comando RETR del protocollo);
  • put: comando che permette di trasferire un file locale nella directory locale al server (comando STOR del protocollo);
  • dir: comando che permette di ottenere la lista dei files nella directory remota (comando LIST del protocollo);
  • quit:  per chiudere la connessione (comando QUIT del protocollo).
     

Si vogliono inoltre implementare i seguenti comandi del protocollo:

  • USER: per acquisire il nome dell'utente;
  • PORT: per comunicare la porta su cui trasferire i dati;
  • CWD:  per cambiare la directory;
  • PWD:  per conoscere la directory di lavoro;
  • DELE: per cancellare un file remoto.

Nel definire l'interazione tra il Client FTP ed il  Server scritto dall'utente, si vogliono osservare le raccomandazioni rfc 959che definiscono il protocollo FTP.

Il client ed il server devono essere dotati di un'interfaccia grafica minimale ed eventualmente di file di log.

Il server, limitatamente alle funzionalità sopra elencate, devono essere in grado di funzionare con almeno un client FTP fornito dal sistema operativo sia esso Windows o Linux.
 

Progetto numero 2.  Realizzazione di un Sistema  SMTP.


L'obiettivo di tale progetto la realizzazione di un  sistema  Cleint/Server SMTP e POP3 per l'invio e la ricezione della posta elettronica secondo gli standard  RFC0821 e RFC1939.

 Il server SMTP dovra' svolgere le seguente funzionalita':

  1. Ricevere ed inviare i messaggi da un' applicazione  per la posta elettronica di un utente locale.
  2. Salvare i messaggi provenienti da altri utenti diretti ad un utente locale in un file secondo il formato richiesto dal server POP3 RFC1939.

 Il server POP3 dovrà:

  1. sotto richiesta del client accedere ai file di posta degli utenti locali e renderli disponibili al client.

Il Client SMTP dovra' svolgere le seguenti funzionalita':

  1. Risolvere gli indirizzi degli utenti di destinazione attraverso l'accesso al DNS locale.
  2. Recapitare i messaggi al SMTP server dell'utente di destinazione.
  3. Prelevare i messaggi dal server POP3 e visualizzarli.

Il client ed il server devono essere dotati di un'interfaccia grafica minimale ed eventualmente di file di log.

I server, limitatamente alle funzionalità sopra elencate, deve essere in grado di funzionare con un client di posta elettronica fornito dal sistema operativo sia esso Windows o Linux.

Progetto numero 3.  Visualizzazione del grafo di Internet

L'obiettivo di questo progetto è l'implementazione di un applicazione che consenta di visualizzare il grafo di Internet. L'applicazione riceve in input un insieme di pagine P , ed un intero N. 

Ad ogni pagina p appartenente a P è associato un thread che, a partire da p, ne naviga tutti i link e così ricorsivamente fino ad un profondità pari a N, costruendo il grafo G_p che rappresenta tale navigazione. Nel grafo G_p esiste un arco che collega due nodi a e b se e solo se esiste un link tra la pagina a e b. 

Il grafo complessivo G ottenuto dall'unione dei sottografi G_p deve essere visualizzato opportunamente. I nodi  di G_p, che rappresentano pagine HTML possono essere cliccati e navigati.

Progetto numero 5. Slides interattive.

L'obiettivo di tale progetto e la  realizzazione di un'applicazione che consenta la visualizzazione di slides su un server Web. Inoltre l'applicazione dovrà gestire un insieme di FAQ (Domande frequenti) relative a ciascuna slide.

specifiche: l'applicazione sarà utilizzata da tre gruppi di utenti

  1. Question (Q): sono coloro che pongono le domande sulle slides. Per inserire una domanda è sufficiente cliccare sul bottone Q presente su ogni slide.
  2. Answer (A): sono coloro che rispondono alle domande. Per rispondere ad una domanda è sufficiente cliccare sul bottone A presente su ogni slide e qunidi specificare a quale domanda rispondere (Q1, Q2 ...)
  3. Super User (SU):  Sono coloro che inseriscono le slides e danno l'approvazione alla pubblicazione delle domande e delle risposte sul sito.

Ciascun gruppo di utenti si dovrà registrare prima di accedere al servizio. Quando un utente del gruppo Q vorrà porre una domanda, cliccherà sul pulsante Q della slide corrispondente, quindi verrà identificato, e solo dopo l'identificazione potrà porre la domanda. La nuova domanda sarà inserita in un Data Base, ma sarà visibile solo dai SU.

Per rendere visibile a tutti la domanda il SU dovrà approvarla esplicitamente.

La stessa procedura vale per le risposte. Ovvero una risposta sarà pubblicata solo previa autorizzazione del SU.

Una volta al giorno il SU riceverà via mail l'eventuale notifica circa la presenza di domande/risposte da pubblicare.

Una volta al giorno gli utenti Q, se esplicitamente specificato durante l'inserimento di una domanda, riceveranno l'eventuale notifica di una risposta.

Progettare l'applicazione Web ed implementarla attraverso Servlet in modo che ci sia possibile: registrarsi, navigare attraverso le slides, porre domande e inviare risposte ed infine ricever notifiche secondo le specifiche fornite sopra.

Progetto numero 7.  Proxy HTTP

 

L'obiettivo del progetto è la realizzazione di un proxy HTTP dotato di cache. Tale proxy dovrà ricevere le richieste di comunicazione inoltrate da un browser qualunque (IE, Netscape ...) e verificare se tali richieste sono presenti nella cache locale. Qualora la richiesta sia presente nella cache locale il proxy invierà al browser il contenuto richiesto prelevandolo direttamente dalla cache locale. Qualora la richiesta non sia presente nella cache locale il proxy si connetterà al server specificato, scaricherà il contenuto richiesto, lo inserirà all'interno della cache locale ed infine lo invierà al browser richiedente. La politica di eviction della cache sarà decidibile dall'amministratore. Dovrà essere implementata almeno la politica FIFO. Un file di log manterrà l'elenco di tutte le richieste dal browser specificando se tali richieste saranno state servite dalla cache locale piuttosto che dal server remoto.

Progetto numero 8. Peer-to-peer File Sharing

If you logged into Napster to download a song, here's what happened:

  1. You started the Napster software on your machine. Your machine became a small server able to make files available to other Napster users.
  2. Your machine connected to Napster's central servers. It told the central servers which files were available on your machine. So the Napster central servers had a complete list of every shared song available on every hard disk connected to Napster at that time.
  3. You typed in a query for a song. Let's say you were looking for the song "Roxanne" by The Police. Napster's central servers listed all of the machines storing that song.
  4. You picked a version of the song from the list.
  5. Your machine connected to the user's machine that had that song, and downloaded the song directly from that machine

There are two main similarities between Gnutella and Napster:

  • Users place the files they want to share on their hard disks and make them available to everyone else for downloading in peer-to-peer fashion.
  • Users run a piece of Gnutella software to connect to the Gnutella network.

There are also two big differences between Gnutella and Napster:

  • There is no central database that knows all of the files available on the Gnutella network. Instead, all of the machines on the network tell each other about available files using a distributed query approach.
  • There are many different client applications available to access the Gnutella network.

 

 

Progetto numero 8. SETI distribuito

Un server centrale deve interpretare i dati ricevuti da un'antenna puntata verso lo spazio in cerca di intelligenze. Al fine di rendere più efficiente l'analisi, tali dati vengono suddivisi in frame e inviati a dei client distribuiti che li elaborano in cerca di pattern predefiniti. Quando un client identifica un segnale o parte di esso, deve reinviare il frame al server che lo dovrà catalogare all'interno di un data base.

 

I dati ricevuti dall'antenna sono modellati come sequenze di byte.

Definire una classe JAVA antenna, che generi dati randomicamente e occasionalmente introduca pattern predefiniti che identifichino la presenza di intelligenze.

es: PATTERN 0qwe23, DATI: sdfghjijhgfdfghjk0qwe23gnjhnjmkjhgf

Implementare un'architettura Client/Server per l'analisi di tali dati.

Il server dovrà:

  • registrare i client
  • prelevare i dati dalla classe antenna, suddividerli in frame di lunghezza opportuna e quindi inviarli verso i client. Un frame distinto per ogni client.

                esempio DATI: sdfghjijhgfdfghjk0qwe23gnjhnjmkjhgf      FRAME1:sdfghjijhgfdfghjk0q       FRAME2:  we23gnjhnjmkjhgf

  • immagazinare i dati catalogati ed eventualmenti sincronizzarli.

Il client è un'applet che connettendosi al server riceve i frame, visualizza un grafico associato ai dati ricevuti e li analizza in cerca di opportuni pattern. Quando un pattern viene identificato, il client si connette al server inviando la posizione di  tale pattern nel frame ed una breve descrizione.

ATTENZIONE: La suddivisione in frame potrebbe scindere un pattern in duo o più frames (vedi esempio sopra), quindi i client dovranno essere in grado di riconoscere anche pattern parziali all'inizio o alla fine dei frame e dovranno rimandarli al server che con una successiva elaborazione determinerà se i pattern parziali appartengono effettivamente ad un pattern. In tal senso il server deve mantenere una sincronizzazinoe tra i frames.
