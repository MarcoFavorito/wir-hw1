Lezione 6: Metodologia di progetto

Combina:

  1. Progettazione sistemi informativi
  2. Progettazione di sistemi ipermediali

Riguarda: Contenuto e Navigazione.

Orientata all'uso di XML per la progettazione logica.

NON TRATTA: la presentazione, la definizione dei processi di interfaccia

Fasi della metodologia

  1. Concettuale
  2. Logica
  3. Fisica
  4. Lessicale-Sintattica
  5. Semantica

Le prime tre fasi riguardano la generazione dei documenti, le ultime due riguardano l'elaborazione dei documenti.

Obiettivi

  1. FASE CONCETTUALE: L'obiettivo della prima fase è l'individuazione del dominio di interesse.
  2. FASE LOGICA: Nella seconda fase si individuano le tipologie di documenti di interesse per l'applicazione e si scrivono i relativi DTD.
  3. FASE FISICA: La terza fase riguarda la scrittura dei documenti XML.
  4. FASE LESSICALE-SINTATTICA: Nella quarta fase il documento viene analizzato da un parser per verificarne la conformità allo standard XML e in alcuni casi al suo DTD.
  5. FASE SEMANTICA: Nella quinta fase si interpreta la struttura del documento e se ne elabora il contenuto.

Fase logica: Tipologie di documenti e relazioni

INPUT: Schema ER

OUTPUT: tipologie dei documenti e loro inter-relazioni

Una tipologia di documento è rappresentata da un frammento del diagramma ER in cui viene stabilito un criterio di navigazione e fatta una selezione degli attributi.

  • Individuazione delle tipologie di documento candidate

  • Selezione delle tipologie di interesse

  • Analisi delle gerarchie

  • Individuazione delle relazioni tra tipologie di documenti

Individuazione delle tipologie di documento candidate

  • entità

  • relazioni con le entità che vi partecipano

  • viste

Per ciascuna tipologia che contiene più di una entità occorre definire una entità proprietaria che verrà utilizzata per l'accesso.

Linee guida per la selezione delle tipologie di documenti

  • documenti/viste interessanti

  • specializzazione dei DTD per favorire la ricerca delle informazioni

  • incorporare le entità che corrispondono ad attributi multivalore

  • incorporare le entità che possono essere ricondotte ad attributi strutturati

Esempio di assorbimento di una entità

Gerarchie

  1. definire un unico DTD per tutta la gerarchia

  2. definire una tipologia di documento per ogni foglia

Ad esempio possono essere definite tipologie di documento per 

  • support-staff (che comprende Administartive e Technician)

  • PhD Student

  • Faculty

  • Researcher

La soluzione adottata ha implicazioni sulle fasi successive:

  1. interpretazione unica dei documenti con diverse azioni semantiche

  2. interpretazione di documenti separati

Collegamenti/relazioni  tra le tipologie di documento

  1. Si considerano tutte le relazioni tra le entità che formano le tipologie di documenti

  2. Si selezionano i collegamenti scegliendo per i due versi di ciascuna relazione se rappresentarla tramite un collegamento

Fase logica: Scrittura dei DTD

INPUT: Schema ER e tipologie di documenti

OUTPUT: DTD dei documenti

  1. Costruzione del Root element

  2. Mapping degli attributi

  3. Mapping delle relazioni

  4. Costruzione del content model

Costruzione del Root element

Il root element viene designato dal nome della entità proprietaria.

Ad esempio:

<department> <research-group> <faculty>

Mapping degli attributi

Attributi monovalore non strutturati possono essere modellati da

  • attibuti XML (indipendenti dall'ordine)
  • elementi XML con content PCDATA (ordinati)

Attributi multivalore o strutturati

  • elementi XML con content PCDATA

Mapping delle relazioni

  1. Relazioni interne alla tipologia di documento -> attributi
  2. Relazioni tra tipologie di documento diverse -> link

Si usa il linguaggio XLink che consente di specificare i collegamenti tra i tag dei documenti.

Costruzione del content model

Ordinamento degli elementi derivati dagli attributi  e dalle relazioni interne.

Cardinalità degli elementi

Esempio

<!element faculty(name,address,phone-number,birthday,e-mail,home-page, student-hour,interested-in+,member-of*,issues*,teaches*)+>

Attributi
NAME
<!element name EMPTY> 
<!attlist name
key id #REQUIRED
type CDATA #FIXED "string"
lengthmax CDATA #FIXED "30">

BIRTHDAY
<!element birthday EMPTY> 
<!attlist birthday
type CDATA #FIXED "data"
day CDATA #REQUIRED
month CDATA #REQUIRED
year CDATA #REQUIRED>

STUDENT-HOUR
<!element student-hour EMPTY> 
<!attlist student-hour
day (lun|mar|mer|gio|ven) #REQUIRED
hour CDATA #REQUIRED>

Relazioni

INTERESTED-IN
<!element interested-in EMPTY> 
<!attlist interested-in
name ID #REQUIRED>

TEACHES
<!element teaches PCDATA> 
<!attlist teaches
key CDATA #FIXED "yes"
xml:link CDATA #FIXED "simple"
href CDATA #REQUIRED>

Fase fisica

INPUT: Archivio, Base Dati (XML/Tradizionale), DTD dei documenti

OUTPUT: Documento XML (Statico/Dinamico)

Si utilizzano strumenti diversi a seconda di:

  • sorgente dei dati

  • realizzazione statica/dinamica

Fase fisica (caso statico)

I dati vengono estratti al momento della creazione del sito inizialmente ed inseriti in un documento XML.

  1. Programma di importazione (traduttore).
  2. Estrazione dati dal dbms (tramite connessione col DB o tramite la creazione di un archivio)
  3. Estrazione diretta da archivio XML tramite parser o XML-QL

Se i dati provengono da diverse sorgenti occorre prevedere dei processi di integrazione.

L'output (documento XML) può essere memorizzato:

  1. direttamente nel file system
  2. tramite DBMS (testuale o XML)

Fase fisica (caso dinamico)

I dati vengono estratti al momento della creazione del documento XML che avviene sulla base delle richieste dell'applicazione ed inseriti in un documento XML dinamico.

L'estrazione può avvenire:

  1. Lettura dell'archivio
  2. Connessione con DBMS
  3. Estrazione diretta da archivio XML tramite parser o XML-QL

Nel caso dinamico:

  1. il processo di creazione delle pagine è controllato dall'applicazione che richiede il documento XML
  2. si complica notevolmente il problema di gestire dinamicamente l'integrazione dei dati.

Fase lessicale-sintattica

INPUT: Documento XML e relativo DTD

OUTPUT:Analisi del documento

Strumenti: Parser XML, Parser ad hoc

Il parser può essere validante, ma la fase lessicale sintattica perse di importanza quando il documento XML viene generato automaticamente.

Fase semantica

INPUT: contenuto del documento XML

OUTPUT: Elaborazione del documento

Strumenti: Parser XML, XSL, DOM, SAX, Parser ad hoc

L'obbiettivo di questa fase è la determinazione delle azioni semantiche per:

  1. il mapping dei documenti sulle unità di visualizzazione

  2. struttura navigazionale dell'applicazione

  3. gestione delle transazioni e dell'interazione con l'utente
